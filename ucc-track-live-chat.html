Service & Process Design - UCC Program

Live Chat

Designing the live chat experience from MVP to full channel maturity

Vera Crisci (Service Design) | June 2026

Track Scope

What this track covers

The Live Chat track is the most mature UCC workstream. It introduced a new channel for ABU travellers on AWS Connect, starting with web-only English MVP and progressively expanding to app, AITS entry and relaxed eligibility criteria

SPD has been involved from the start - defining the experience, designing journeys per phase and shaping the channel as it scales.

SPD focus: Experience definition, detailed user journeys (traveller and agent) per phase, service blueprinting, requirements translation into user stories, agent UX design ownership and post-launch monitoring.

Key attributes

  • First channel built natively on AWS Connect
  • Real-time synchronous
  • Web + app (iOS/Android) entry points
  • AITS chatbot as triage layer before agent handover
  • PEGA agent desktop integration

Status

Live Chat is scaling by expanding the agent pool within MNL5. Multi-site expansion is on hold until critical bugs are fixed and key missing capabilities are delivered.

Experience Assessment

Key experience observations

Needs attention

No channel strategy in place

There is no formal channel strategy defining when a traveller should be steered to Live Chat versus other channels. SPD proposes a three-tier model (self-serve first, Live Chat for time-sensitive and single-session authenticated contacts, async for everything else) but this has not been confirmed at program level. Without it, volume allocation risks being driven by operational capacity rather than what actually fits the traveller's need.

Delivering well

AITS warm handover with full context

Since Phase 2, the AITS conversation summary is automatically sent as the first message when a chat is handed over to an agent. This means agents already know the traveller's intent, topic and reservation details before they start typing. The traveller does not need to repeat themselves, which is one of the most common friction points in support interactions.

Needs attention

Progressive journey design per phase

Each delivery phase has a documented journey that clearly shows what is available and what is not. Experience principles govern design decisions across all phases. However, experience trade-offs are frequently driven by technical complexity and effort rather than design rationale. Several items have been deprioritised, with operations absorbing the gap through workarounds - a pattern that is not sustainable at scale.

Needs attention

Requirements-first, experience-second

When SPD joined the program, the requirements had already been listed. Experience definition came after, not before. Ideally, the target experience should be defined first and requirements derived from it, otherwise there is a risk of building features that are technically correct but do not connect into a coherent traveller journey.

Needs attention

Routing and continuity gaps

Currently agents handle a single chat at a time with availability-only routing - there is no skill-based matching. If a connection drops, the traveller cannot reconnect to the same agent. Follow-up after a chat ends is only possible via email or phone, as there is no conversation inbox yet. This creates a fragmented experience when issues span multiple interactions.

Needs attention

MVP restrictions narrowing reach

The channel launched with tight eligibility criteria (desktop web, English, non-AITS entry point, one reservation only). Phase 2 expanded this significantly with app support, AITS entry, dynamic eligibility and multi-reservation. However, non-English languages and file attachments remain excluded until BAU, which limits the channel's usefulness for a large portion of travellers.

Not yet available

Translation and async as future scale unlocks

Real-time translation would decouple agents from language matching, meaning any agent could serve any market. Async messaging would introduce persistent conversation threads with push notifications, so travellers are not forced into a synchronous session when their issue does not require one. Together, these two capabilities would unlock reach across all markets and remove dead ends from the experience.

Not yet available

Unified agent desktop and fluid AITS

The Day 3 vision is voice, chat and messaging unified in one agent workspace with knowledge integrated into the agent's workflow. It also targets a single-window AITS-to-Live Chat transition (currently two separate windows, though the handover works well on app). Today, agents work in Pega with the AWS Connect chat as a separate side panel rather than a natively integrated experience.
