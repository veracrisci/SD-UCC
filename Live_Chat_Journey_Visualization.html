Live Chat Experience Journey

Live Chat Journey & Capability Readiness View - CCaaS AWS Amazon Connect

This journey maps the traveler and agent experience for Live chat across the different delivery phases, showing capability readiness against the listed requirements and prioritisation assigned as of June 4th 2026. Each phase represents an incremental release of the live chat channel on AWS Amazon Connect, from MVP launch through to full operational maturity (labelled BAU) and beyond.

Owner: Vera Crisci - Service Design

Day 0 / Day 1 - MVP Journey 31.03 / 14.04

The first live chat experience for Booking.com travellers, launched on web for the MNL5 site. Travellers reach live chat through the Help Center after triage, with hardcoded eligibility in English only. Agents operate on a single-chat model with basic tooling on Pega.

Web only English only Help Center entry Hardcoded eligibility Single chat per agent VNO6 Day0 MNL5 Day1
Stage 01
📞
Entry & Contact
Traveller reaches live chat via Help Center triage on web.
Traveller experience

Opens Help Center on web, navigates triage questions about their booking issue, and is offered live chat if eligible (English, active reservation, within operating hours). If chat is unavailable, sees phone or email as alternatives.

Agent experience

Has no visibility at this stage. Is unaware of incoming travellers until a chat is assigned. Cannot influence which travellers reach them or see queue volume.

Available
  • Web Help Center triage entry
  • Self-service options before LC
  • Fallback to phone/email if LC unavailable
  • Kill switch via feature flag
Not yet available
  • App entry (iOS/Android)
  • AITS chatbot handover to LC
  • PO reservation access
Stage 02
🪪
Identification
Traveller authenticated; reservation context retrieved.
Traveller experience

Is automatically authenticated via their logged-in session. Does not need to provide booking details - the system retrieves their reservation context behind the scenes.

Agent experience

Receives the traveller's booking number, topic selected during triage, and prior CS interaction history on Pega. Does not see AITS context or multiple reservations.

Available
  • Hardcoded eligibility (context, EN, hours)
  • Agent scheduling check
  • Reservation context retrieval
  • Booking #, topic, prior CS interactions
Not yet available
  • Dynamic eligibility via API
  • Vertical-agnostic identification
  • AITS context handover
Stage 03
🏷️
Classification
Topic from triage determines categorisation; English only.
Traveller experience

Has already selected a topic during Help Center triage. Waits to be connected without any further input required. The topic selection determines how the request is categorised.

Agent experience

Receives the triage topic and reservation type as context. Does not get AI-generated summaries, intent signals, or sentiment analysis before the chat starts.

Available
  • Topic from HC triage steps
  • Language detection (EN only)
  • Reservation type context
Not yet available
  • AITS intent/sentiment
  • AI-generated summary for agent
  • Smart channel steering
  • Queue-aware classification
Stage 04
📋
Prioritization
Traveller queued; routed to available agent in ring-fenced group.
Traveller experience

Is placed in queue and sees a longest-wait-time estimate. Receives system welcome messages while waiting. Cannot leave and get notified - must stay on the page.

Agent experience

Receives the next chat from the ring-fenced queue based on availability only. Has no control over assignment, cannot see queue depth, and cannot be selected by Genius level or skill.

Available
  • Route to available agent (ring-fenced group)
  • Queue expectation (longest wait)
  • Welcome/system messages
  • Acceptable latency matching
Not yet available
  • Route by Genius/attributes
  • Escalation to senior
  • Push notifications while queued
Stage 05
🛎️
Service
Real-time text interaction on web; single chat per agent.
Traveller experience

Chats in real-time on web with typing indicators and read receipts. Can click URLs shared by the agent. Can leave the tab and return within the session timeout. Cannot send images, files, or receive push notifications.

Agent experience

Handles one chat at a time on Pega with pre-defined templates and clickable URLs. Can make outbound calls to partners during the session. Manages inactivity manually - no automated timeout rules. Cannot receive attachments from travellers.

Available
  • Real-time text messaging (web)
  • Typing indicators, read receipts
  • Clickable URLs (plain text)
  • Pre-defined agent templates
  • Outbound calls during chat
  • Case ownership during long processes
  • Leave tab & resume (session timeout)
  • Manual inactivity workaround
Not yet available
  • Route to same agent on reconnection
  • Multi-channel case management
  • Attachments (images, PDF, video)
  • Translations
  • Concurrent chats
  • Push notifications on reply
  • Contextual templates
  • Automated inactivity rules
Stage 06
🔄
Close the Loop
Session ended; transcript downloadable; follow-up via email/phone.
Traveller experience

Session ends and can download the transcript on web. Any follow-up from the agent arrives via email or phone - not in the chat thread. Cannot access past chat transcripts from an inbox.

Agent experience

Closes the session and wraps up the service case in Pega. Handles follow-up work separately via email or phone. Quality is reviewed via Qualtrics. Basic CSAT, AHT, and productivity reports available.

Available
  • Agent close session & wrap up
  • Follow-up via email/phone
  • Follow-up work in Pega
  • Transcript download (web)
  • QM review (Qualtrics)
  • CSAT, AHT, productivity reports
  • GDPR + data retention compliant
Not yet available
  • Async follow-up in thread
  • Inbox transcript access
  • Restart/resume closed chat
  • All standard CS metrics

Pre-BAU* - Hardening Phase 02.06 - 30.06

The hardening phase focuses on security, robustness and operational confidence. Attachments become available with malware scanning, message content is actively filtered and redacted, automated inactivity rules replace the manual workaround, and latency SLAs are validated end-to-end. These enhancements primarily affect Stage 05 (Service) and system-level resilience.

*BAU = Business As Usual - the state where Live Chat is fully handed over to CS Operations as a stable, production-grade channel with no further program-level delivery dependencies.

+ Attachments + Security redaction + Automated inactivity + Latency SLAs confirmed

BAU* - Operational Phase 30.06

Live chat is now a stable, production-grade channel fully handed over to CS Operations. This phase focuses on expanding language support through real-time translations, localising the UI, enabling vertical-agnostic identification, and building toward inbox transcript access. Discovery work begins on concurrency, VoIP escalation, and senior escalation paths.
