Service & Process Design - UCC Program

Written Channels

Migrating the highest-volume support channels without degrading the experience for 29.8M contacts per year

Vera Crisci (Service Design) | June 2026

Track Scope

What this track covers

The Written Channels track aims to migrate ABU's email, messaging and web form channels from Intercom (and related legacy systems) into Amazon Connect. This is categorically different from Live Chat and Voice: these are not new channels and not a bounded pilot. They are the channels that 29.8M travellers and partners already use, with existing threads, open cases and active expectations.

Any degradation, like a traveller who stops receiving updates mid-case, a reply thread that breaks, an agent who loses context at handover, will affect contacts already in progress. The traveller does not know a migration happened, they only know the service got worse.

SPD focus: We recommend designing AWS Connect to its full CX potential rather than doing a like-for-like migration of the current setup, as such an approach would move the existing experience into new infrastructure without improving it (re-platforming the 49.2% CSAT ceiling rather than addressing it). Amazon Connect's native capabilities, such as intelligent routing, automated classification and proactive updates, can help deliver a better experience already at launch, with no Pega changes required and within the current program scope. Currently, four program-level decisions are pending before the requirements sign-off can proceed.

Key attributes

  • Existing channels under live load - not new, not bounded
  • Highest CS volume (29.8M contacts, 81% in scope)
  • Worst CSAT performance (Email 49.2%, Messaging 52.4%)
  • Fragmented architecture (Intercom, SAPI, Camunda, Pega, GMU, 80+ aliases)
  • Journey design is risk management, not preference

Current State

The failure mechanism

The 81.8% email re-contact rate is not an anomaly. It is the system working exactly as designed. Three evidence layers converge on the same diagnosis:

90%

No-reply templates

Pega CS email templates are no-reply. When a traveller replies to cs-noreply@booking.com, they receive a hard 'address not found' error. The sender address begins with 'cs' rather than 'noreply', which reads as a human inbox and actively encourages replies that cannot be received.

74.8%

Repeat contacts

Request Information cases are repeat contacts. Of 34,527 analysed interactions (Mar-Apr 2026), 25,820 were returning travellers. 15,754 had the same prior topic - meaning the original interaction did not resolve the question. Median gap: 2 days. 7,175 returned same day.

54%

Status lookups

Agent-handled contacts are pure status lookups - automatable today without LLM complexity. These contacts consume agent capacity without requiring human judgement.

The mechanism: Incomplete first contact, no proactive follow-up, and a traveller waiting in silence until the absence of a response becomes sufficient reason to contact again. A platform migration that treats CSAT as a guardrail rather than a goal will not close this gap.

Context

Why this track differs from Live Chat and Voice

Live Chat introduced a new channel - travellers had no prior experience to compare against, no existing threads to preserve. Voice in Hungary replaces a platform (Cisco to Connect) in a bounded, single-market context where service journeys are already established. Neither touched an existing channel under live load with active customer expectations.

Written channels are categorically different. Email and messaging are not new and not bounded. When this program cuts over a flow, it replaces something already running with active human expectations on the other side. This is the condition that makes journey-anchored design a risk management requirement, not a preference.

Key entry points

F13 Help Centre Contact Form 10.4M/yr
F01 Guest Email 5.6M/yr

Both are IFD decisions. Designing them without the IFD principle is actively building an entry experience that does not meet it.

Gap Analysis

Capability gaps not reflected in current requirements

The Async UCC Requirements and Capabilities Inventory contains 29 functional requirements across 8 capability stages. It maps what systems must do. It does not map what travellers, agents or partners need to experience at each stage.

Contact entry and acknowledgement

Exists: Identity attachment at intake

Auto-acknowledgement content, tone and design have no owner. Traveller's first impression of the new system is not a defined deliverable.

Outbound / notifications

Exists: Noted as goal area; delivery path described

Template design, milestone triggers and no-reply policy have no CX design. 90% no-reply ratio is inherited by default.

Agent desktop and context

Exists: Pega loads with context

"Full context" is not defined. Knowledge base is absent from requirements. AI co-pilot is absent from scope entirely.

Automation and deflection

Exists: Email automation bot integration

Scope of automatable intents undefined. Handoff quality undefined. 54% of contacts are status lookups - automatable without LLM complexity.

Measurement

Exists: Contact records, unified schema

CSAT trigger point, RCR measurement methodology and quality monitoring are not defined. CSAT classified as a guardrail, not a success metric.

Routing

Exists: Routing and priority rules

Routing block explicitly marked as placeholder pending WFM/Ops review. 476 skills not rationalised. Priority policy not defined.

Recommendation

Three paths identified

Path 1

Technical migration

Like-for-like. Preserve current routing, case model and outbound setup. Low delivery risk. Re-platforms the 49.2% CSAT ceiling into new infrastructure. Connect's native CX capabilities not engaged.

Path 2 - Recommended

Full CX potential

Connect designed with native CX capabilities fully engaged. No Pega changes required. Achievable within current scope. Materially better experience at launch. Requires journey design input before requirements sign-off.

Path 3

+ Pega optimisation

Path 2 extended with Pega service journey improvements: template redesign, knowledge base surfacing and agent process optimisation. Full North Star potential. Outside current scope.

Four program-level decisions are needed before requirements sign-off can proceed: (1) Treat CSAT, RCR (re-contact rate) and AHT as success metrics the Program is accountable for, not just guardrails to avoid breaching. (2) Confirm that user journey design is a required input before requirements are signed off. (3) Use the same capability language across the program and the North Star Blueprint so requirements map to the target architecture. (4) Decide which approach the program will follow for Written Channels (see paths above).
